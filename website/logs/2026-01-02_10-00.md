1→It seems to me there will be two main AI coding classes, which are probably subclasses/specializations of the static versus dynamic typing divide.
     2→
     3→The static typers will become verifiers. Their code will focus on making sure the robot can work by itself for as long as possible, aided by types, tests, well-defined specs, complex agent setups, and so on.

The core insight rings true to me: there are two fundamentally different strategies for working with AI-generated code, and they optimize for different things.

**Verifiers** build scaffolding the AI can bounce off of — the feedback loop is between the AI and formal artifacts (tests, types, specs). The human steps back and lets the system converge.

**Readers** keep themselves in the loop — the feedback is between human comprehension and AI output. The human stays close, reading everything, catching problems through understanding rather than automated checks.

Where I'd push back slightly: I'm not sure the static/dynamic typing correlation is as strong as you suggest. The actual divide might be more about *trust disposition* than typing preference. Some static typers are extremely hands-on readers (Rust programmers often care deeply about understanding every line). Some dynamic typers happily write extensive test suites and let CI be the verifier.

The deeper variable might be: "Do you trust formal artifacts to catch problems, or do you trust your own comprehension?" That correlates with typing preference but isn't caused by it.

Also worth noting: the reader strategy has a scaling ceiling. At some point, you can't read everything. The verifier strategy has a setup cost that may never pay off for smaller projects. So project size might force people toward one end regardless of preference.

But as a crude division for thinking about how people will relate to AI coding tools — yes, I think it captures something real.

Several additions worth considering:

**The reader ceiling is moveable.** Tooling pushes it higher — good diffs, smart summarization, AI-assisted review of AI output. You can read more if the reading is made easier. This matters because it means the "readers don't scale" point is true but not fixed. Investment in reading-assistance tooling could extend the approach further than raw human cognition allows.

I agree with the core claim: verifiers scale into agent societies in a way readers can't. The logic is sound — if the feedback loop is AI ↔ formal artifacts rather than AI ↔ human, then:

1. One agent can run longer without human intervention
2. Multiple agents can interface through formal contracts (specs, APIs, typed interfaces)
3. Agent A's verified output becomes Agent B's verified input
4. Verification at boundaries enables trust without comprehension
5. Composition yields capabilities beyond any single agent

This is how organizations work. No CEO reads every line of code. Trust flows through contracts, interfaces, accountability structures. Agent societies would mirror this — specialization, trade, contracts, emergent complexity.

**But I'd push back on the reader framing being "completely useless."** I think it transforms rather than disappears.

Consider: verification at the component level doesn't guarantee system-level properties. An agent society might have every agent passing its tests while the emergent behavior is wrong, unexpected, or misaligned with actual goals. Goodhart's law scales too — optimize for specs, not intentions, and the gap widens as complexity increases.

Someone needs to watch the system's *behavior*, not its code. Not verifying against formal specs, but judging whether the whole thing is going in the right direction. This is a reading function — but reading outputs, decisions, trajectories rather than code.

The reader moves up the abstraction stack:
- **Before:** Reading code to ensure quality
- **After:** Reading agent behavior to ensure alignment with goals that can't be fully formalized

This maps to human organizations again. Managers don't read code, but they read reports, review outcomes, make judgment calls about direction. The oversight function doesn't disappear — it changes what it's looking at.

The alternative — fully autonomous agent societies with no human oversight — is possible but raises hard questions about steering and accountability. Verification can tell you "this agent did what it was told" but not "this was the right thing to tell it to do."

> So in your view, the reader class now, which reads a single agent's output, will eventually need to evolve into reading the results of agent's actions? Could you imagine how I could apply this to making games, for instance?

Yes, and for games specifically this is interesting because games are so taste-dependent.

**Current state (reader):** You work with a single agent, read the code it produces, make taste calls about feel and juice, tight feedback loop at the code level.

**Future state (overseer):** You can't read all the code because there's too much, or because multiple agents produced it. But you can *play the game*. The game itself becomes what you read.

The feedback loop shifts:
- **Now:** prompt → code → read code → adjust prompt
- **Later:** direction → agent work → play build → give direction

This is actually how larger studios already work. A game director doesn't read every line of code. They play builds, watch playtests, say things like "this doesn't feel punchy enough" or "the difficulty curve is off." The team translates that into concrete changes.

With agent societies, the "team" is agents:
- Physics agent (tuning bounce, collision feel)
- Particle/juice agent (impact effects, screen shake)
- Level generation agent (layouts, pacing)
- Balance agent (damage numbers, spawn rates)
- Art agent (sprites, visual coherence)

Agents verify each other at interfaces. You play the result and give direction.

**Concrete workflow for your games:**

1. You establish core feel directly (reader mode — you're in the code, tight loop, this is where taste lives)
2. Once core feel is locked, agent society handles content generation (levels, enemy variants, item combinations)
3. Agents verify against specs you've defined (balance constraints, difficulty curves, spawn rules)
4. You play generated content, judge whether it *feels* right
5. Direction flows back: "these levels feel too cramped" → agents iterate

**The hybrid maps to your existing philosophy.** From CLAUDE.md: "Get moment-to-moment gameplay working first (physics, core mechanics). Then surrounding systems." The core is reader territory — you stay close, read code, control feel. The surrounding systems are verifier territory — specs, generation, agent composition.

**What you'd need for this to work:**

- Fast iteration (playable builds quickly, hot reloading)
- Vocabulary for feel that agents understand ("punchier", "floatier", "more weight")
- Specs that capture your taste where possible (though some taste resists specification)
- Trust calibration — knowing which agent outputs to spot-check vs. trust

**The risk:** feel-language is lossy. "Make it punchier" might get misinterpreted ten times before an agent gets it right. The reader approach is slower per-change but higher fidelity. The overseer approach is faster throughput but risks drift from your taste.

So the reader doesn't disappear — they become a director. Reading the game instead of the code. Making taste calls at the experience level instead of the implementation level. The skill shifts from "can I understand this code" to "can I articulate what's wrong with this feel and trust agents to fix it."

> Doesn't this is imply that, as a reader, I should be focused on creating verifier-like specs for all these future agents you mentioned, so that once the technology is easier to use they already have enough to go from?

Interesting strategic question. Let me push back a bit.

**The case for yes:** Specs encode taste durably. When agent tech matures, you're ready. The work compounds. Writing specs also clarifies your own thinking.

**The case for no:**

1. **You don't know the format.** Future agent systems might want specs in a shape you can't predict. Formal specs written now might need translation anyway.

2. **Premature formalization.** The best specs emerge from working systems, not upfront design. You discover what matters by building, not by speculating about what will matter.

3. **Taste evolves.** Specs lock in current preferences. Your sense of "good feel" in 2028 might differ from 2026. Formal specs become debt if they encode stale taste.

4. **Time trade-off.** Hours spent writing specs are hours not spent shipping games. The games themselves are more valuable than preparation for hypothetical future workflows.

5. **You're already doing it.** CLAUDE.md, ANCHOR.md, your code patterns, your engine design — these *are* proto-specs. They encode taste in a reader-friendly format. They're not formal, but they're extractable. When agents get good enough, you (or an agent) can formalize them.

**My actual take:** Don't write formal specs. Do document your taste, but keep it lightweight and example-based:

- Video clips of "this feels right"
- Prose descriptions ("collisions should feel like X, not Y")
- Reference moments from other games
- Patterns and conventions in CLAUDE.md style

This is raw material for specs without being specs. It stays flexible. It evolves with you. It's useful *now* for working with single agents like me. When agent societies arrive, the formalization can happen then — possibly by an agent that reads your documentation and extracts specs from it.

**The deeper point:** Your taste, living in your head, might be more defensible than specs. Specs can be copied, shared, commoditized. The judgment that produces good specs — that's harder to replicate. Staying in reader mode keeps that judgment sharp. Premature shift to verifier mode might atrophy the very skill that makes your games good.

Ship games. Document taste lightly as you go. Formalize when there's concrete demand, not before.

## HTML Details/Summary Elements for Collapsible Sections

The `<details>` element is the element to contain the content we want to show and hide. The `<summary>` element is a caption to describe what is hidden, in an FAQ section this might be the question and the answer is hidden in the details element until it is expanded.

### How It Works

The `<details>` element is a container for content that can be expanded or collapsed by the user. The `<summary>` element defines a visible label or heading for the collapsible section. Clicking this toggles the visibility of the content inside `<details>`.(no content)This functionality works natively in all modern browsers and can be styled with CSS.

### Basic Example

```html
<details>
  <summary>What is HTML?</summary>
  <p>HTML stands for HyperText Markup Language. It is the standard language for creating web pages.</p>
</details>
```

When the user clicks the summary text, the paragraph becomes visible. Clicking again hides it.

### Key Benefits

You can create expandable/collapsible sections with pure HTML elements—no npm libraries needed!(no content)You can toggle the details element open and closed with JavaScript if needed in your use case by setting the `open` attribute on it.(no content)Keyboard users can navigate and toggle them using Tab and Enter.(no content)`<details>` and `<summary>` offer a semantic and accessible way to create collapsible content without any JavaScript. They're perfect for simple interactions like FAQs or expandable instructions.
